# SQLite Evidence Census v1

Date: 2026-06-26

## Purpose

SQLite Evidence Census v1 adds read-only reporting for the additive SQLite evidence mirror. It helps answer whether Momentum Hunter has enough mirrored data for offline analysis, whether candidate fields are complete enough to trust, and whether query performance is acceptable before any future SQLite read-mode adoption.

This is measurement only. It does not change scoring, readiness, alerts, outcomes, trade planning, or file authority.

## Reports

### SQLite Query Benchmark

Command:

```powershell
.\.venv\Scripts\python.exe -B -m momentum_hunter.sqlite_benchmarks
```

Outputs:

```text
MomentumHunterData/data/reports/sqlite-query-benchmark-latest.json
MomentumHunterData/data/reports/sqlite-query-benchmark-latest.md
```

Measures:

- candidate history query
- capture session counts
- alert outcome counts
- minute-bar symbol counts
- evidence-run counts
- latest system-status events
- provider quality checks
- user-state mirror counts

## Safety Rules

- SQLite remains additive and non-authoritative.
- File-based sources remain authoritative.
- Reports are read-only against SQLite.
- No raw captures are mutated.
- No user-authored state is mutated.
- No synthetic data is written to production evidence stores.
- No scoring, readiness, alert, outcome, or trade-planning logic is changed.

## Current Interpretation

These reports are intended to make bottlenecks visible:

- Low completed alert count means optimization remains locked.
- Missing relative volume or freshness fields should be treated as data-quality limits.
- Query timings should stay comfortably below the slow-query threshold before SQLite-powered UI read models expand.
