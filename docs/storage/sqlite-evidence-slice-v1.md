# SQLite Evidence Slice v1

SQLite Evidence Slice v1 mirrors opportunity alerts and alert outcomes into SQLite without changing the file-based runtime. `opportunity-alerts.json` remains the active derived source of truth.

## Safety Contract

- Do not mutate raw captures.
- Do not rewrite `opportunity-alerts.json`, `alert-outcome-update-status.json`, `opportunity-minute-bars.json`, CSVs, or Markdown reports during import.
- Do not change scoring math, readiness rules, alert generation, outcome classification, scanner thresholds, trade-planning rules, broker behavior, or automation behavior.
- Keep SQLite additive until conflict handling, backups, and source-of-truth cutover rules are designed.

## Source Files

| Area | Source file | Owner modules | Current write behavior | SQLite table |
| --- | --- | --- | --- | --- |
| Opportunity alerts | `MomentumHunterData/data/opportunity-alerts.json` | `momentum_hunter.opportunity_alerts.load_alerts`, `save_alerts` | Alert engine and outcome updater update this derived evidence store | `opportunity_alerts` |
| Alert outcomes | Embedded in each alert inside `opportunity-alerts.json` | `momentum_hunter.opportunity_alerts`, `momentum_hunter.alert_outcome_updater` | Outcome updater matures pending alerts to completed or terminal unscorable states | `alert_outcomes` |
| Outcome update status | `MomentumHunterData/data/alert-outcome-update-status.json` | `momentum_hunter.alert_outcome_updater.save_update_report`, `load_update_report` | Latest updater status file, overwritten by updater runs | Not imported in this slice |
| Minute bars | `MomentumHunterData/data/opportunity-minute-bars.json` | `momentum_hunter.alert_outcome_updater.load_minute_bars`, `save_minute_bars` | Derived market-data cache for outcome validation | Deferred next slice |
| Alert performance analytics | `opportunity-alerts.json` | `momentum_hunter.alert_performance` | Reads alerts and writes disposable performance reports | Uses SQLite later after validation |

## Alert ID Strategy

Alert IDs are generated by `momentum_hunter.opportunity_alerts.stable_alert_id(symbol, timestamp, alert_type)` and stored in the JSON alert as `alert_id`.

SQLite uses `alert_id` as the primary key for `opportunity_alerts`.

Uniqueness rule:

```text
opportunity_alerts.alert_id
```

If duplicate alert IDs appear inside one source file, the importer keeps the first and reports `DUPLICATE_ALERT_ID_IN_SOURCE`.

## Outcome ID Strategy

Alert outcomes are one-to-one with opportunity alerts in the current derived store, so SQLite uses:

```text
alert_outcomes.alert_id
```

as both the primary key and the alert/outcome join key. No separate outcome ID is created in this slice because the source file does not have multiple outcome versions per alert.

## Timestamp Conventions

- Alert `timestamp` remains the event timestamp from the source alert.
- `imported_at` records when SQLite mirrored an inserted or changed row.
- `updated_at` records when the SQLite mirror row last changed.
- Source file hash uses SHA-256 and is stored as `source_alerts_hash`.
- Timestamps are stored as ISO-8601 text with timezone offset when available.

## Missing Data Behavior

The importer is defensive:

- Missing optional quote fields import as `NULL`.
- Missing optional RVOL, volume, bid/ask, spread, stop, targets, or catalyst fields do not block import.
- Missing `opportunity-alerts.json` produces a warning and zero imported rows.
- Missing alert IDs are skipped and reported as warnings.
- The importer does not attempt to calculate missing outcomes.

## Unscorable Outcome Handling

Terminal data-quality outcomes are mirrored exactly:

- `outcome.status = UNSCORABLE_OUTCOME`
- `outcome.classification = UNSCORABLE_MISSING_ENTRY_PRICE`
- `outcome.classification = UNSCORABLE_MISSING_MARKET_DATA`
- `outcome.classification = UNSCORABLE_INCOMPLETE_EVIDENCE`
- `outcome.classification = UNSCORABLE_INVALID_TIMESTAMP`

Unscorable rows are not converted to pending, success, failure, noise, or late. They remain visible in SQLite counts and import reports but do not count as completed outcomes.

## Import / Upsert Rules

Imports can be run repeatedly.

- New alerts insert into `opportunity_alerts`.
- Changed alert rows update safely.
- Unchanged alert rows are skipped.
- New embedded outcomes insert into `alert_outcomes`.
- Matured outcomes update safely, such as `PENDING` to `SUCCESSFUL`.
- Unchanged outcomes are skipped.
- Existing JSON/CSV/Markdown files are not changed.

The importer stores the full source alert/outcome JSON in SQLite `source_json` fields for auditability.

## CLI

Run only the evidence slice:

```powershell
.\.venv\Scripts\python.exe -m momentum_hunter.sqlite_migration --slice evidence
```

Outputs:

```text
MomentumHunterData/data/reports/sqlite-evidence-import-latest.json
MomentumHunterData/data/reports/sqlite-evidence-import-latest.md
```

## Validation Rules

The import report checks:

- Source alert count equals source alert rows in SQLite.
- Source outcome count equals source outcome rows in SQLite.
- Pending, completed, and unscorable classifications survive import.
- Repeated imports skip unchanged rows.
- Source files are not modified.

## Import Risks

| Risk | Mitigation |
| --- | --- |
| JSON schema drift | Use `load_alerts()` so existing defensive parsing remains the gate. |
| Duplicate alert IDs | Detect and warn; do not create duplicate SQLite rows. |
| Outcome maturing after first import | Upsert updates changed rows by primary key. |
| Permanent unscorable rows being mistaken for pending | Preserve `UNSCORABLE_OUTCOME` and classification separately. |
| SQLite becoming accidental source of truth | Keep CLI/report wording explicit that JSON remains active. |
| Minute-bar evidence being needed for deeper audits | Defer `minute_bars` to the next slice rather than partially importing it here. |

## Recommended Next Slice

`minute_bars` should be the next SQLite evidence slice. Minute bars support alert outcome validation, chart/evidence replay, and future market-structure analysis, but they need retention and provider-quality policy before they become a reliable normalized store.
